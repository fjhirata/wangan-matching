# 湾岸マッチング（プロトタイプ / フェーズ1）

性格・価値観 × 経済条件で、湾岸エリアのタワーマンション約47棟から
あなたにベストな「居住スタイル」と「物件TOP3」を提案する、提案型マッチングサイト。
MBTI風のエンタメ性と、データに基づく実益を両立し、**SNS拡散**と**LINE個別相談への誘導**を狙う。

> ### データの実装状況（全49棟・2つの実DBを結合）
> ✅ **主DB（成約履歴 5,843件）**：坪単価（直近3ヶ月の成約移動平均／1取引1票）・値上がり率（資産性）・成約面積中央値（広さ）・成約件数・代表間取り。
> ✅ **従DB（物件スペック 49棟）**：駅徒歩・築年・耐震・総戸数・共用施設（プール/ジム/バー/サウナ/コンビニ/内廊下/オール電化）。
> → 物件名（NFKC正規化＋空白除去）で結合し、**5軸スコアをすべて実データから算出**：
>   資産性=値上がり率、広さ=成約面積、施設=共用施設、立地=駅徒歩、居住性=築年/耐震/設備。
> ※ 坪単価の移動平均の基準はデータ内最新成約月（現状 2026/05）。直近3ヶ月に成約が無い棟のみ窓を6→12ヶ月へ自動拡大し、月数と件数をUIに明示。
> ※ 従DBに築年月日が無い2棟は築年を非表示（捏造しない）。スプレッドシートは**読み取りのみ・未編集**。

## なぜこの構成か（この環境に最適化）

このMacには Node.js / npm が無いため、**追加インストール不要で即動く**よう
**「Python標準ライブラリのみのバックエンド ＋ 静的フロントエンド」** で構築しています。

```
ブラウザ (静的SPA)  ──fetch──▶  Python APIサーバ (http.server)  ──▶  データ層 db.py
  static/                          backend/                         data/*.json（将来 Google Sheets / REINS）
```

将来 React/Next.js 等へ移行する場合も、APIの形（`/api/diagnose` 等）はそのまま流用できます。

## 起動方法

```bash
cd wangan-matching
python3 backend/server.py
# → ブラウザで http://localhost:8000 を開く
```

ポート変更： `PORT=9000 python3 backend/server.py`

手動でデータを作り直す： `python3 tools/build_mansions_from_db.py`

### データ自動更新（スプレッドシート更新の自動反映）

スプレッドシートは主DBが月1回更新される。サーバは起動中、**裏で定期的に両シートを取得し直して自動反映**する（再起動不要）。

- 間隔： `REFRESH_HOURS`（既定 12時間）。例 `REFRESH_HOURS=6 python3 backend/server.py`
- 即時反映： シート更新直後に `GET/POST /api/refresh` を叩く（ブックマーク可）。例 `curl http://localhost:8000/api/refresh`
- 保護： `REFRESH_TOKEN=任意の文字列` を設定すると `/api/refresh?token=…` でのみ更新可。
- 取得失敗時は**直前のデータを維持**（壊れない）。取得は**読み取りのみ**でシートは編集しない。
- 現在のデータ鮮度は `GET /api/health` とフッターの「データ更新 …」で確認可。

> ⚠️ 「月次更新を完全に無人で反映」させるには、サーバを**常時起動（サーバ/クラウドにデプロイ）**しておく必要があります（ローカル起動中のみ自動更新が動作）。常時稼働させればこのループが月次更新を自動で取り込みます。

## ユーザー体験（UXフロー）

| STEP | 内容 | 実装 |
| --- | --- | --- |
| 1 | 前提条件入力（年齢・世帯年収・希望の広さ㎡） | `static`（スライダー＋予算ライブ計算） |
| 2 | 価値観診断20問・5段階回答（1問ずつ自動送り） | `static` ＋ `data/questions.json` |
| 3 | タイプ判定／5軸レーダー／物件TOP3 | `engine.diagnose()` |
| 4 | LINE個別相談へ誘導＋X（旧Twitter）シェア | 結果画面のCTA |

### ロジック（engine.py）

- **予算**：総予算 = 世帯年収 × 7（上限目安）。許容坪単価 = 総予算 ÷ 希望坪数（㎡÷3.30578）。
- **5軸**：20問の回答（±2）を 居住性 / 資産性 / 施設充実度 / 立地 / 広さ の各軸 0–100 に正規化。逆転項目対応。
- **タイプ**：ユーザーの5軸ベクトルに最も近い「理想ベクトル」を持つタイプを採用（全13種）。
- **TOP3**：予算内を優先し、価値観の重み付きスコアが高い順に提示。予算内が3件未満なら近い物件で補完（「予算オーバー」表示＝LINE相談の動機に）。

## API（バックエンド）

| メソッド | パス | 内容 |
| --- | --- | --- |
| GET | `/api/health` | 死活確認＋データ鮮度（source / generatedAt / 経過時間） |
| GET | `/api/config` | 軸定義・回答スケール・年収倍率・LINE URL・データ鮮度 |
| GET | `/api/questions` | 診断20問 |
| GET | `/api/mansions` | 物件一覧（実DBから集計） |
| GET | `/api/types` | 購入タイプ一覧（全13種） |
| POST | `/api/diagnose` | `{age, income, areaSqm, answers}` → 診断結果（型・5軸・TOP3） |
| GET/POST | `/api/refresh` | スプレッドシートから即時再取得・反映（`?token=…` で保護可） |

`POST /api/diagnose` リクエスト例：
```json
{ "age": 38, "income": 1200, "areaSqm": 70,
  "answers": { "1": 2, "2": 1, "3": 2, "...": 0, "20": -1 } }
```

## データ更新・差し替え手順

**データ最新化**：2つのスプレッドシートを取得し直して再集計するだけ（**取得＝読み取りのみ。シートは編集しません**）。
```bash
rm data/raw/transactions.csv data/raw/spec_master.xlsx   # 最新を取り直す場合
python3 tools/build_mansions_from_db.py                  # 主DB(CSV)＋従DB(xlsx)を取得→mansions.json 再生成
```

- 主DB＝成約CSV（gid指定）、従DB＝スペックxlsx（「検索・ソート情報」タブを自動判別）。
- 物件名で結合。表記揺れは NFKC 正規化＋空白除去で吸収（49/49一致を確認済み）。
- スコアの重み・係数を調整したい場合は `build_mansions_from_db.py` の `score_*()` を編集。

上位ロジック・フロントは変更不要です（`db.py` ＋ ビルダーが唯一の接続点）。
